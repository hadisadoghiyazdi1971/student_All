# سامانه ارجاع هوشمند نامه‌های دانشگاهی

این نسخه برای استفاده واقعی در محیط دانشگاه طراحی شده است. سامانه نامه را دریافت می‌کند، حوزه علمی را با ساختار واقعی دانشگاه تطبیق می‌دهد، مقاله‌های مرتبط را به صورت معنایی بازیابی می‌کند، اعضای هیئت علمی را با دلایل قابل توضیح رتبه‌بندی می‌کند و یک پیش‌نویس قابل ویرایش از نامه ارائه می‌دهد.

## جریان پردازش

```text
نامه متنی / PDF / Word / تصویر
        ↓
استخراج متن و نرمال‌سازی فارسی
        ↓
KeyBERT با تعداد و آستانه کنترل‌شده
        ↓
تطبیق معنایی با taxonomy واقعی دانشگاه
        ↓
Ollama + qwen3:8b برای تحلیل ساختاریافته
        ↓
بازیابی معنایی مقاله‌ها با BAAI/bge-m3
        ↓
اتصال مقاله‌ها به نویسندگان و رتبه‌بندی
        ↓
نمایش دلیل انتخاب هر فرد
        ↓
تولید پیش‌نویس پیشنهادی نامه
```

## حالت‌های تحلیل

دو گزینه مستقل در رابط کاربری وجود دارد:

1. **فقط KeyBERT:** استخراج کلیدواژه و تحلیل قانون‌محور. در صورت غیرفعال بودن Ollama، پیش‌نویس نامه با یک قالب امن تولید می‌شود.
2. **فقط Ollama:** متن نامه و حوزه‌های کاندید taxonomy مستقیما به مدل داده می‌شوند.
3. **حالت ترکیبی:** خروجی KeyBERT و حوزه‌های کاندید همراه متن اصلی به Ollama داده می‌شوند و مدل خروجی نهایی را تولید می‌کند.

حداقل یکی از دو گزینه باید فعال باشد.

## کنترل کلمات کلیدی

عدد انتخاب‌شده در رابط کاربری **سقف خروجی** است و نه تعداد اجباری. روند استخراج شامل موارد زیر است:

- تولید تعداد بیشتری کاندید نسبت به خروجی نهایی
- استفاده از MMR برای جلوگیری از عبارات بسیار مشابه
- حذف عبارت‌های عمومی و اداری
- حذف موارد پایین‌تر از حداقل امتیاز
- حذف عبارت‌های تکراری و هم‌پوشان
- بازگرداندن تعداد کمتر در صورتی که عبارت معتبر کافی وجود نداشته باشد

## استفاده از فایل‌های Excel دانشگاه

دو فایل زیر در `data/taxonomy` قرار دارند:

- `university_37_field_names_only.xlsx`: نام‌های استاندارد حوزه‌ها
- `university_37_medical_fields_only.xlsx`: دانشکده، گروه آموزشی، رشته یا گرایش و تعداد افراد

داده‌ها قبل از استفاده نرمال می‌شوند. تفاوت‌هایی مانند `ي/ی`، `ك/ک`، نیم‌فاصله، فاصله اضافه و برخی شکل‌های تکراری حذف می‌شوند. نام اصلی برای نمایش حفظ می‌شود.

سامانه فقط کلیدواژه‌ها را با نام حوزه مقایسه نمی‌کند. امتیاز حوزه از ترکیب معنایی متن کامل، کلمات KeyBERT و در حالت Ollama خلاصه مدل ساخته می‌شود.

## بازیابی مقاله‌ها

TF-IDF نسخه قبلی با جست‌وجوی معنایی جایگزین شده است. امتیاز مقاله ترکیبی از موارد زیر است:

- شباهت embedding متن و مقاله
- هم‌پوشانی محدود کلمات تخصصی برای حفظ اصطلاحات دقیق

embedding مقاله‌ها روی دیسک کش می‌شود. هر زمان داده مقاله یا نام مدل تغییر کند، cache به صورت خودکار بازسازی خواهد شد.

## رتبه‌بندی افراد و Explainability

امتیاز هر مقاله برای نویسنده از این عوامل ساخته می‌شود:

- ارتباط مقاله با نامه
- نقش نویسنده اول
- نقش نویسنده مسئول
- تعداد استناد
- FWCI

برای جلوگیری از برتری غیرمنصفانه افراد دارای مقاله‌های بسیار زیاد، فقط شواهد برتر هر فرد با کاهش وزن تدریجی جمع می‌شوند.

برای هر فرد این موارد نمایش داده می‌شوند:

- تعداد مقاله‌های مرتبط
- میانگین ارتباط
- تعداد نقش نویسنده اول و مسئول
- مقاله‌های شاخص
- کلیدواژه‌های مشترک
- توضیح متنی علت پیشنهاد

## ذخیره آخرین تنظیمات

تنظیمات در SQLite و مسیر `runtime/settings.db` ذخیره می‌شوند. مواردی مثل تعداد مقاله، تعداد افراد، تعداد کلمات کلیدی، مدل‌ها و فعال بودن KeyBERT/Ollama بعد از restart باقی می‌مانند.

کلید `APP_SETTINGS_PROFILE` مشخص می‌کند تنظیمات متعلق به کدام پروفایل است. در نسخه فعلی این پروفایل در سطح deployment مشترک است. برای تنظیمات مستقل هر کاربر در محیط چندکاربره باید شناسه کاربر سامانه احراز هویت دانشگاه به این کلید متصل شود.

## پیش‌نویس پیشنهادی نامه

پیش‌نویس پس از تحلیل و رتبه‌بندی تولید می‌شود و نامه اصلی را تغییر نمی‌دهد. کاربر می‌تواند موارد زیر را انتخاب کند:

- بازنویسی نامه فعلی یا ساخت نامه جدید
- لحن رسمی، اداری، علمی یا کوتاه
- مخاطب فرد اول پیشنهادی، گروه آموزشی یا واحد ارجاع

خروجی شامل عنوان، مخاطب، متن قابل ویرایش، نکات بهبود و اطلاعات ناقص است و به صورت TXT یا Word دانلود می‌شود. مدل اجازه ندارد نام، تاریخ، شماره دانشجویی، اطلاعات تماس یا ادعای جدید اختراع کند.

## ساختار داده‌های اصلی

## نصب محلی

Python 3.11 پیشنهاد می‌شود.

```bash
python -m venv .venv
```

ویندوز:

```powershell
.venv\Scripts\activate
pip install -r requirements.txt
copy .env.example .env
```

لینوکس:

```bash
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
```

## آماده‌سازی Ollama

Ollama را نصب و اجرا کنید:

```bash
ollama pull qwen3:8b
ollama serve
```

سپس وضعیت سرویس و داده‌ها را بررسی کنید:

```bash
python startup_check.py
```

## اجرای برنامه

```bash
streamlit run app.py
```

در اولین تحلیل، مدل `BAAI/bge-m3` دانلود می‌شود و نمایه حوزه‌ها و مقاله‌ها ساخته می‌شود. برای سرور دانشگاه بهتر است نمایه را قبل از شروع سرویس بسازید:

```bash
python scripts/prebuild_embeddings.py
```

## اجرای تست‌ها و بررسی کیفیت کد

وابستگی‌های توسعه را نصب کنید:

```bash
pip install -r requirements-dev.txt
pytest -q
ruff check .
```

## Docker

```bash
docker compose up -d --build
```

پس از اجرای کانتینر Ollama مدل را دانلود کنید:

```bash
docker compose exec ollama ollama pull qwen3:8b
```

فایل‌های اصلی دانشگاه باید پیش از build یا اجرا در پوشه `data` قرار گرفته باشند.

## الزامات استقرار واقعی

این موارد باید قبل از بهره‌برداری رسمی انجام شوند:

1. قرار دادن برنامه پشت HTTPS و SSO یا reverse proxy دانشگاه
2. محدود کردن دسترسی پوشه داده و cache به حساب سرویس
3. تهیه نسخه پشتیبان از داده‌ها و `runtime/settings.db`
4. ممنوع بودن ثبت متن کامل نامه در logها
5. تعریف دوره نگهداری فایل‌ها و نتایج مطابق سیاست حریم خصوصی دانشگاه
6. ارزیابی با مجموعه‌ای از نامه‌های برچسب‌خورده و تنظیم آستانه‌های `keyword_min_score` و `field_min_score`
7. تعریف مسیر بررسی انسانی برای confidence پایین یا نبود نتیجه
8. ثبت نسخه داده، مدل و تنظیمات مورد استفاده در هر release
9. آزمون بار، مانیتورینگ CPU/GPU/RAM و زمان پاسخ
10. بازبینی امنیتی و حقوقی پیش از اتصال به سامانه مکاتبات دانشگاه

## محدودیت مهم
